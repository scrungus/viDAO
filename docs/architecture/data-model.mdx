---
title: "Data model"
description: "Prisma schema walkthrough — users, videos, sessions, monthly totals, and payouts."
---

The data model lives in `apps/web/prisma/schema.prisma`. Six models, three enums. The shape is deliberately boring: nothing is reconstructed from events, and every authoritative value is a column.

## Enums

- `UserRole` — `VIEWER` or `CREATOR`. Gates upload and dashboard access.
- `AttestationType` — `HARDWARE` or `SOFTWARE`. Defaults to `SOFTWARE`. Only `HARDWARE` users accrue weighted watch time.
- `PayoutStatus` — `PENDING` or `DISTRIBUTED`. Tracks whether the on-chain `distribute` call completed for a period.

## User

The central entity. One row per authenticated account.

| Field | Notes |
|---|---|
| `email` | Unique. Sourced from Privy. |
| `walletAddress` | Unique. Privy embedded wallet address. |
| `webauthnCredentialId` | Optional. Set when the user enrolls a passkey. |
| `webauthnAttestationType` | Defaults to `SOFTWARE`. Flipped to `HARDWARE` on successful passkey enrollment. |
| `subscribedUntil` | `DateTime?`. A user is "subscribed" when this is `> now`. |
| `role` | `VIEWER` by default; `CREATOR` role is set manually. |

## Video

Creator-owned content.

| Field | Notes |
|---|---|
| `creatorId` | FK to `User`. Only users with `role = CREATOR` can own videos. |
| `videoPath` | Path on local disk (`VIDEO_UPLOAD_DIR`), served via Next.js. |
| `thumbnailUrl` | Best-effort thumbnail generated by `ffmpeg` during upload. |

## WatchSession

One row per `/api/watch/start` call. Mutated by `/api/watch/heartbeat` and `/api/watch/end`.

| Field | Notes |
|---|---|
| `userId`, `videoId` | The session owner and content. |
| `startedAt` | Set once, on `start`. |
| `lastHeartbeatAt` | Updated on every valid heartbeat — and also on gap-too-large events (to re-anchor the next heartbeat). |
| `weightedSeconds` | Accumulates only when the user is `HARDWARE`-attested, actively subscribed, and under the monthly cap. |

## MonthlyWatchTotal

Aggregated view used by the monthly cap check. Composite primary key `@@id([userId, month])` where `month` is `YYYY-MM`.

| Field | Notes |
|---|---|
| `weightedHours` | `Float`. Running total for the user's current month. Capped at 50. |

The heartbeat endpoint reads this inside a transaction before incrementing `WatchSession.weightedSeconds`.

## PayoutPeriod

One row per run of the payout cron. Created by `runPayoutCron` in `apps/web/src/server/payout-cron.ts`.

| Field | Notes |
|---|---|
| `periodStart`, `periodEnd` | Time window aggregated for this period. |
| `totalPoolUsdc` | Snapshot of `pool_balance()` at the moment the cron ran. |
| `status` | `DISTRIBUTED` if the on-chain call succeeded, `PENDING` otherwise. |

## CreatorPayout

Child of `PayoutPeriod`. One row per creator with non-zero watch hours in the period.

| Field | Notes |
|---|---|
| `watchHours` | Creator's share of total watch hours (hours, not seconds). |
| `sharePercentage` | `watchHours / totalHours`. |
| `usdcAmount` | The amount passed to `distribute(creators, amounts)`. |
| `txHash` | On-chain transaction hash, populated on success. |

## Relationships

```text
User ─────── 1:N ──────── Video
  │                         │
  │                         └── 1:N ── WatchSession ── N:1 ── User (same)
  │
  └── 1:N ── MonthlyWatchTotal
  │
  └── 1:N ── CreatorPayout ── N:1 ── PayoutPeriod
```

A `User` can simultaneously be a viewer (via `WatchSession`, `MonthlyWatchTotal`) and a creator (via `Video`, `CreatorPayout`). The role check is only about upload/dashboard permissions.
